# 启动部署指南

## 部署模式概述

系统支持两种部署模式：

| 模式 | 说明 | 适用场景 |
|---|---|---|
| **单体模式** | 单进程同时运行 QUIC + HTTP | 开发调试、单机小规模部署 |
| **拆分模式** | QUIC 和 HTTP 分离为独立进程 | 生产环境、独立扩缩容、容错隔离 |

---

## 环境要求

- Rust 1.82+ (2024 edition)
- PostgreSQL 数据库
- Redis 服务
- TLS 证书（放置于 `config/ssl/` 目录）
  - `fullchain.pem` — 证书链
  - `privkey.pem` — 私钥

## 配置文件

`config/app_config.toml`：

```toml
[database]
url = "postgres://admin:password@localhost:5432/postgres"

[redis]
url = "redis://:password@localhost:6379/"

[server]
address = "0.0.0.0:8443"

[quic_server]
address = "0.0.0.0:4433"
server_name = "SERVER_1"

[internal_quic_server]
address = "127.0.0.1:4434"
server_name = "INTERNAL_SERVER_1"
```

### 环境变量占位符

配置文件中支持 `${VAR_NAME}` 占位符，启动时自动替换为对应的环境变量值：

```toml
[database]
url = "${DATABASE_URL}"

[redis]
url = "${REDIS_URL}"
```

配合 `.env` 文件使用：

```env
DATABASE_URL=postgres://admin:password@localhost:5432/postgres
REDIS_URL=redis://:password@localhost:6379/
```

## 方式一：单体模式

所有服务运行在同一个进程中，启动最简单。

### 编译

```bash
cargo build --release
```

### 启动

```bash
cargo run --release
```

或直接运行编译产物：

```bash
./target/release/rust_my_app
```

### 监听端口

| 端口 | 服务 |
|---|---|
| 8443 | HTTPS API |
| 4433 | QUIC 外网（客户端连接） |
| 4434 | QUIC 内网（服务间通知） |
| 9562-9565 | P2P UDP 打洞 |

### 特点

- 启动简单，一条命令
- 资源共享，内存占用较小
- 一个服务崩溃会影响全部
- 无法对 QUIC / HTTP 独立扩缩容

---

## 方式二：拆分模式

QUIC 服务端和 HTTP API 端分离为两个独立进程，互不依赖。

### 编译

```bash
# 编译 QUIC 服务端
cargo build --release -p quic_service --bin quic_server

# 编译 HTTP API 端
cargo build --release -p api --bin api_server
```

### 启动

先启动 QUIC 服务端（对外暴露 QUIC 端口），再启动 HTTP API 端：

```bash
# 终端 1 — QUIC 服务端
cargo run --release -p quic_service --bin quic_server

# 终端 2 — HTTP API 端
cargo run --release -p api --bin api_server
```

或直接运行编译产物：

```bash
# QUIC 服务端
./target/release/quic_server

# HTTP API 端
./target/release/api_server
```

### 启动顺序

QUIC 服务端和 HTTP API 端**没有严格的启动顺序要求**：

- 先启动 QUIC：Redis 注册为幂等操作，HTTP 端稍后启动不受影响
- 先启动 HTTP：内网 QUIC 注册失败时降级为 warn 日志，不会退出，后续自动恢复

推荐先启动 QUIC 服务端，确保客户端连接就绪，再启动 HTTP API 端。

### 监听端口

| 进程 | 端口 | 服务 |
|---|---|---|
| quic_server | 4433 | QUIC 外网（客户端连接） |
| quic_server | 4434 | QUIC 内网（服务间通知） |
| quic_server | 9562-9565 | P2P UDP 打洞 |
| api_server | 8443 | HTTPS API |

### 特点

- QUIC / HTTP 独立进程，互不影响
- 可独立扩缩容（HTTP 可水平扩展，QUIC 按需部署）
- 一个服务挂掉不会拖垮另一个
- 需要分别管理两个进程

## systemd 部署（Linux 生产环境）

### 单体模式

`/etc/systemd/system/rust-im.service`：

```ini
[Unit]
Description=Rust IM Service
After=network.target postgresql.service redis.service

[Service]
Type=simple
User=im
WorkingDirectory=/opt/rust_my_app
ExecStart=/opt/rust_my_app/target/release/rust_my_app
Restart=always
RestartSec=5
Environment=RUST_LOG=info

[Install]
WantedBy=multi-user.target
```

### 拆分模式

`/etc/systemd/system/rust-im-quic.service`：

```ini
[Unit]
Description=Rust IM - QUIC Service
After=network.target redis.service
Before=rust-im-api.service

[Service]
Type=simple
User=im
WorkingDirectory=/opt/rust_my_app
ExecStart=/opt/rust_my_app/target/release/quic_server
Restart=always
RestartSec=5
Environment=RUST_LOG=info

[Install]
WantedBy=multi-user.target
```

`/etc/systemd/system/rust-im-api.service`：

```ini
[Unit]
Description=Rust IM - HTTP API Service
After=network.target postgresql.service redis.service

[Service]
Type=simple
User=im
WorkingDirectory=/opt/rust_my_app
ExecStart=/opt/rust_my_app/target/release/api_server
Restart=always
RestartSec=5
Environment=RUST_LOG=info

[Install]
WantedBy=multi-user.target
```

启用服务：

```bash
# 单体
sudo systemctl enable --now rust-im

# 拆分
sudo systemctl enable --now rust-im-quic rust-im-api

# 查看状态
sudo systemctl status rust-im-quic rust-im-api
```

## 优雅关闭

两个 binary 均支持通过 `SIGINT`（Ctrl+C）或 `SIGTERM` 优雅关闭：

- **quic_server**：收到信号后，关闭 QUIC Endpoint（现有连接收到 `b"server shutdown"`），等待 100ms 让 accept loop 退出，标记服务为 Stopped
- **api_server**：actix-web 内置优雅关闭，等待进行中的请求完成后退出

```bash
# systemd 优雅关闭
sudo systemctl stop rust-im-quic
sudo systemctl stop rust-im-api
```

## 模式选择建议

| 场景 | 推荐模式 |
|---|---|
| 本地开发调试 | 单体模式 |
| 小型部署（< 1000 并发） | 单体或拆分均可 |
| 中型部署（1000-10000 并发） | 拆分模式 |
| QUIC 和 HTTP 需要独立扩缩容 | 拆分模式 |
| 需要 QUIC 高可用（多节点） | 拆分模式，配合 Redis 集群路由 |
