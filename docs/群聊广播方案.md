# 群聊广播方案

## 1. 目标

基于现有的集群路由架构，实现**读扩散**模型的群聊消息广播功能。要求：

- 一条群消息只需 **1 次 DB 写入**
- 消息投递链路尽可能少 Redis/DB 查询
- 各节点只查自己的 `DashMap`，不跨节点
- 支持在线成员 QUIC 推送 + 离线成员上线后拉取

---

## 2. 核心原则

### 2.1 群聊用广播，单聊用 hash 路由

```
群聊 → 1 条消息 → N 个接收者 → 所有节点都有活干 → 广播
单聊 → 1 条消息 → 1 个接收者 → 其他节点白干活 → hash 精确路由
```

互不干扰，各用各的最优方案。

### 2.2 一次查成员列表，多处复用

接收节点从 Redis/DB 查询一次全量群成员列表后，**随广播消息一起下发**，其他节点不再重复查询：

```
接收节点 → 查成员列表（1 次 Redis）
        → 将消息 + 成员列表打包广播到所有内网节点
             → 各节点直接用成员列表 + 本机 DashMap，0 次 Redis
```

---

## 3. 消息流程

### 3.1 完整时序

```
发送者 A                   Node 1 (接收节点)         Node 0 / Node 2     Redis / DB
    │                             │                       │                  │
    │── 群消息 ──────────────────►│                       │                  │
    │                             │                       │                  │
    │                             │── 查群成员列表 ──────────────────────────►│
    │                             │◄── ["A","B","C","D"] ────────────────── │
    │                             │                       │                  │
    │                             │── 校验 A 是否在群内 ── │                  │
    │                             │   通过 ✓              │                  │
    │                             │                       │                  │
    │                             │── 广播到 Node 0 ─────►│                  │
    │                             │   {msg, ["A","B.."]}  │                  │
    │                             │                       │── 查本机 DashMap │
    │                             │                       │   B→在线→投递 ✓  │
    │                             │                       │   (A,C,D 不在)   │
    │                             │                       │                  │
    │                             │── 广播到 Node 2 ─────►│                  │
    │                             │   {msg, ["A","B.."]}  │                  │
    │                             │                       │── 查本机 DashMap │
    │                             │                       │   D→离线→跳过    │
    │                             │                       │   (A,B,C 不在)   │
    │                             │                       │                  │
    │                             │── 本机处理             │                  │
    │                             │   查本机 DashMap       │                  │
    │                             │   A(自己)→跳过         │                  │
    │                             │   C→在线→投递 ✓       │                  │
    │                             │                       │                  │
    │                             │── 持久化群消息 ────────────────────────► │
    │◄── ACK ─────────────────── │                       │                  │
```

### 3.2 关键说明

| 步骤 | 说明 |
|------|------|
| **查成员列表** | 由接收节点执行，缓存命中走 Redis，未命中走 DB |
| **校验** | 确认发送者 A 是群成员，否则直接拒绝 |
| **广播** | 通过内网 QUIC 发送到除自己外的所有节点 |
| **本机处理** | 不走网络，与广播的`InternalGroupBroadcast`结构体共用同一处理函数 |
| **持久化** | 由接收节点写入 DB，**只需 1 条** |

pub enum BroadcastType {
    GroupText,
    GroupImage,
    GroupFile,
    GroupNotification,
}

### 5.1 接收节点处理入口

```rust
/// 接收客户端群消息后的处理入口
pub async fn handle_group_msg_from_client(
    group_msg: GroupQuicMsg,
    internal_config: &InternalQuicConfig,
    connections: &ConnectionsMap,
) -> anyhow::Result<()> {
    // 1. 序列化群消息体（粘包协议格式）
    let msg_bytes = serialize_group_msg(&group_msg)?;

    // 2. 从 Redis/DB 查全量群成员列表（1 次查询）
    let all_members = get_group_members_cached(&group_msg.group_uuid).await?;

    // 3. 校验发送者是否在群内
    if !all_members.contains(&group_msg.send_user) {
        return Err(anyhow!("发送者不在群成员列表中"));
    }

    // 4. 构造广播消息，带上成员列表
    let broadcast = InternalGroupBroadcast {
        broadcast_type: match group_msg.msg_type {
            10 => BroadcastType::GroupText,
            11 => BroadcastType::GroupImage,
            12 => BroadcastType::GroupFile,
            _ => BroadcastType::GroupNotification,
        },
        group_uuid: group_msg.group_uuid,
        msg_bytes,
        sender: group_msg.send_user,
        all_members,
        source_node: internal_config.server_index,
        timestamp: group_msg.timestamp,
        broadcast_id: group_msg.nano_id,
    };

    // 5. 获取所有内网节点地址，向其他节点广播
    let nodes = get_all_internal_node_addresses().await?;
    for (node_index, addr) in &nodes {
        if *node_index == internal_config.server_index {
            continue;
        }
        // 即使某个节点失败也不影响整体
        let _ = send_internal_group_broadcast(*addr, &broadcast).await;
    }

    // 6. 本机处理（不走网络，直接调用本地逻辑）
    process_group_broadcast_local(&broadcast, connections).await?;

    // 7. 持久化消息（只需写 1 条）
    save_group_message_to_db(&group_msg.nano_id, &group_msg).await?;

    Ok(())
}
```

### 5.2 各节点收到广播后的处理

```
quic_service/src/msg_service/group_msg_service.rs (同文件)
```

```rust
/// 各节点收到内网 QUIC 广播后的处理
/// 用广播中携带的成员列表 + 本机 DashMap 投递，不查任何外部存储
pub async fn process_group_broadcast(
    broadcast: &InternalGroupBroadcast,
    connections: &ConnectionsMap,
) -> anyhow::Result<()> {
    for member in &broadcast.all_members {
        // 跳过发送者自己
        if *member == broadcast.sender {
            continue;
        }

        // 直接查本地 DashMap——在就在，不在就跳过
        let conn = find_online_connection(member, connections);
        if let Some(conn) = conn {
            let mut send = conn.open_uni().await?;
            send.write_all(&broadcast.msg_bytes).await?;
            send.finish().await?;
        }
    }

    Ok(())
}

/// 查本机 DashMap 找用户的在线连接
fn find_online_connection(
    user_uuid: &str,
    connections: &ConnectionsMap,
) -> Option<quinn::Connection> {
    for platform in [PC_PLATFORM, MOBILE_PLATFORM] {
        let key = format!("{}:{}{}{}{}",
            platform,
            REDIS_QUIC_SERVERS,
            user_uuid,
            REDIS_SPLIT,
            ConnectionType::Text,
        ).to_uppercase();

        if let Some(entry) = connections.get(&key) {
            return Some(entry.conn.clone());
        }
    }
    None
}
```

/// 内网 QUIC 路由入口
/// 根据消息类型分发到不同的处理路径
pub async fn route_internal_request(
    request: &[u8],
    connections: &ConnectionsMap,
    server_index: u32,
) -> Result<Vec<u8>> {
    // 尝试反序列化为群聊广播
    if let Ok(broadcast) = bincode::deserialize::<InternalGroupBroadcast>(request) {
        process_group_broadcast(&broadcast, connections).await?;
        return Ok(bincode::serialize(&InternalQuicResponse::ok())?);
    }

    // 尝试反序列化为单聊路由请求
    if let Ok(msg) = bincode::deserialize::<InternalQuicRequest>(request) {
        let resp = route_single_chat_request(&msg, connections, server_index).await?;
        return Ok(bincode::serialize(&resp)?);
    }

    Err(anyhow!("未知的内网 QUIC 请求类型"))
}

## 7. 去重处理

### 7.1 为什么需要去重

在某些异常场景下可能出现重复广播：

- 节点收到广播后回传
- 网络抖动导致重试
- 节点刚恢复，收到历史广播

### 7.2 实现方案

```rust
use dashmap::DashSet;
use std::sync::Arc;

pub struct BroadcastDedup {
    set: Arc<DashSet<String>>,
}

impl BroadcastDedup {
    pub fn new() -> Self {
        let set = Arc::new(DashSet::new());

        // 每 60 秒清空一次（广播 ID 只短时间有效）
        let cloned = set.clone();
        tokio::spawn(async move {
            loop {
                tokio::time::sleep(Duration::from_secs(60)).await;
                cloned.clear();
            }
        });

        Self { set }
    }

    /// true = 首次见到（可以处理）
    /// false = 已处理过（跳过）
    pub fn try_process(&self, broadcast_id: &str) -> bool {
        self.set.insert(broadcast_id.to_string())
    }
}
```

### 7.3 集成到广播处理

```rust
use once_cell::sync::Lazy;

static DEDUP: Lazy<BroadcastDedup> = Lazy::new(BroadcastDedup::new);

pub async fn process_group_broadcast(
    broadcast: &InternalGroupBroadcast,
    connections: &ConnectionsMap,
) -> anyhow::Result<()> {
    if !DEDUP.try_process(&broadcast.broadcast_id) {
        return Ok(()); // 已处理过，跳过
    }

    // ... 正常投递逻辑 ...
}
```

## 8. 离线消息同步

### 8.1 未读标记

在 `group_member` 表维护 `last_read_msg_id`，请求群消息历史时以此为偏移量：

```sql
SELECT * FROM group_message_record
WHERE group_uuid = #{group_uuid}
  AND id > #{last_read_msg_id}
ORDER BY id ASC
LIMIT 100;
```

### 8.2 用户上线时同步

```rust
/// 用户上线后，拉取所有群的未读消息
pub async fn sync_offline_group_messages(
    user_uuid: &str,
    connections: &ConnectionsMap,
) -> anyhow::Result<()> {
    let groups = get_user_groups(user_uuid).await?;

    for group in groups {
        let unread = GroupMessageRecord::select_unread(
            &group.group_uuid,
            group.last_read_msg_id,
        ).await?;

        for msg in unread {
            let msg_bytes = serialize_group_msg_as_bytes(&msg)?;
            if let Some(conn) = find_online_connection(user_uuid, connections) {
                let mut send = conn.open_uni().await?;
                send.write_all(&msg_bytes).await?;
                send.finish().await?;
            }
        }
    }

    Ok(())
}
```

## 11. Redis 缓存策略

### 11.1 群成员列表缓存

**TTL = -1（永不过期）+ 手动失效**，不做 TTL 自动过期。

原因：群成员列表的变更入口完全受控（HTTP API），一致性要求高。

| 方式 | 评估 |
|------|------|
| TTL = -1 + 手动失效 | 实时一致，成员退群后立即不被推送 |
| TTL = 300s | 最坏 5 分钟脏窗口，退群成员仍收到消息 |

### 11.2 失效时机

在以下 HTTP API 中统一删除缓存：

| API | 操作 | Redis 动作 |
|-----|------|-----------|
| 入群 | `insert_group_member()` | `DEL group:members:{group_uuid}` |
| 退群 | `delete_group_member()` | 同上 |
| 踢人 | `update_member_status(踢出)` | 同上 |
| 解散群 | `update_group_status(解散)` | 同上 |

```rust
pub async fn get_group_members_cached(group_uuid: &str) -> anyhow::Result<Vec<String>> {
    let cache_key = format!("group:members:{}", group_uuid);

    if let Some(members) = redis_get::<Vec<String>>(&cache_key).await? {
        return Ok(members);
    }

    let members = fetch_group_members_from_db(group_uuid).await?;

    redis_set_ex(&cache_key, &members, -1).await?;

    Ok(members)
}

pub async fn invalidate_group_member_cache(group_uuid: &str) -> anyhow::Result<()> {
    let cache_key = format!("group:members:{}", group_uuid);
    redis_del(&cache_key).await?;
    Ok(())
}
```

### 11.3 故障恢复

如果缓存数据因意外丢失（如 Redis 重启），下次查询自动回源 DB 重新写入，无需手动干预。

## 12. 集群场景总结

| 维度 | 做法 | 说明 |
|------|------|------|
| **群成员列表** | Redis 缓存，TTL = -1 + 手动失效 | 所有群成员变更 API 均需删除缓存 |
| **成员列表获取** | 接收节点查 1 次 | 随广播下发，其他节点不复查 |
| **在线判定** | 各节点查本地 DashMap | 0 次网络开销，0 次 Redis |
| **节点间通信** | 内网 QUIC 广播 | 1 条消息达到所有节点 |
| **消息持久化** | 接收节点写 DB | 1 条记录 |
| **去重** | broadcast_id + DashSet | 防网络抖动/重试导致的重复广播 |
| **离线同步** | last_read_msg_id + 上线拉取 | 读扩散标准做法 |
| **扩缩容** | 无影响 | 新节点启动后自动回源 DB |
| **单聊路由** | hash % N + Redis 兜底 | 保持现有方案不变，与群聊互不干扰 |
