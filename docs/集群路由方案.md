# IM 集群路由方案

## 1. 目标

将现有的单机 IM 系统升级为**多节点集群**，实现：

- QUIC 网关节点**无状态**，只做消息转发 + ACK
- 消息持久化交由消息队列异步处理（暂不实现，留 TODO）
- 节点间通信采用 **新流 + 请求/响应** 模式，避免粘包处理
- 节点自动发现，无需手动配置对端列表

---

## 3. 节点标识

### 3.1 当前问题

[config_str.rs](../crates/entity/src/config_str.rs#L10) 中 `SERVER_NAME` 是硬编码常量 `"SERVER_1"`，所有实例同名，无法区分节点。

### 3.2 配置文件新增字段

在 `app_config.toml` 的 `[quic_server]` 节下新增 `node_address` 字段：

```toml
[quic_server]
address="0.0.0.0:4433"        # QUIC 监听地址（保持不变）
node_address="192.168.1.10:4433"  # 新增：本节点供其他节点连接的目标地址
```

每个节点部署时各自配置自己的 `app_config.toml`：

| 节点 | `node_address` 值 |
|------|-------------------|
| Node A | `192.168.1.10:4433` |
| Node B | `192.168.1.11:4433` |
| Node D | `192.168.1.20:4433` |

> `address` 是服务端绑定监听的本地地址，通常为 `0.0.0.0:4433`；
> `node_address` 是其他节点用来连接本节点的**可路由地址**，必须是对端可达的 `ip:port`。

### 3.3 读取配置

利用现有配置系统读取该字段：

```rust
// config_str.rs — 新增全局变量，删除硬编码的 SERVER_NAME
use std::sync::OnceLock;
pub static NODE_ADDR: OnceLock<String> = OnceLock::new();
```

```rust
// quic_server.rs — 启动时从配置文件读取
let node_addr = read_global_config!("quic_server", "node_address");
NODE_ADDR.set(node_addr).unwrap();
```

这个值既是**节点身份标识**，也是**其他节点连接此节点的目标地址**，一值两用。

### 3.4 Redis 注册信息的变化

| 改动前 | 改动后 |
|--------|--------|
| `PC:QUIC:SERVER:user1:TEXT` → `"SERVER_1"` | `PC:QUIC:SERVER:user1:TEXT` → `"192.168.1.10:4433"` |
| `PC:QUIC:SERVER:user2:TEXT` → `"SERVER_1"` | `PC:QUIC:SERVER:user2:TEXT` → `"192.168.1.20:4433"` |

## 4. 连接类型区分

QUIC 监听端口只有一个（4433），客户端连接和节点间连接共用同一端口。需要在连接建立时区分：

### 4.1 魔术字节

连接建立后的**第一条流的第一个字节**作为协议标识：

| 魔术字节 | 连接类型 | 后续协议 |
|----------|---------|---------|
| `0x01` | 节点间连接 | 4 字节长度前缀 + bincode 序列化的 `InterNodePayload` |
| `0x10` | 客户端连接 | 剩余字节作为 JSON 解析（现有 `FirstQuicMsg` 逻辑） |

### 4.2 连接分发

```rust
async fn handle_connection(conn: Connection) -> Result<()> {
    // 建立新连接后，第一条流的第一个字节标识连接类型
    let (mut send, mut recv) = conn.accept_bi().await?;
    let mut magic = [0u8; 1];
    recv.read_exact(&mut magic).await?;

    match magic[0] {
        INTER_NODE_MAGIC => {
            // 节点间连接 → 处理对端节点的转发请求
            handle_inter_node_conn(conn).await?;
        }
        CLIENT_MAGIC => {
            // 客户端连接 → 走现有的鉴权 + 消息处理流程
            handle_client_conn(send, recv, address).await?;
        }
        _ => {
            send.finish().await?;
        }
    }
}
```

### 4.3 客户端改动

客户端连接时，第一条流的首字节发送 `0x10`，后续紧跟现有的 JSON 初始化消息。

## 5. 节点间通信协议

### 5.1 设计原则

- **新流 + 请求/响应**：每个转发请求开一条 Quinn `open_bi()`，发送完毕调用 `finish()` 关闭发送端，然后阻塞读取 `RecvStream` 获取响应
- **无粘包**：每个流只包含一个请求 + 一个响应，QUIC 流天然隔离
- **超时控制**：每个转发请求设置 10 秒超时

### 5.2 数据协议

```
请求方向（发送方 → 接收方）:
┌──────────┬──────────────┬──────────────────────────────┐
│  0x01    │  4 字节长度   │    bincode 序列化的           │
│ (魔术字节)│  (大端 u32)  │    InterNodePayload           │
└──────────┴──────────────┴──────────────────────────────┘

响应方向（接收方 → 发送方）:
┌──────────────────────────────────────┐
│   bincode 序列化的 ForwardResult     │
│   (读至 stream 关闭, read_to_end)    │
└──────────────────────────────────────┘
```

### 5.3 数据结构

```rust
/// 节点间转发请求 - 客户端消息的路由信息
struct InterNodePayload {
    msg_id: String,           // 消息 ID
    text_type: u16,           // 消息类型（文本/图片/文件...）
    raw: Vec<u8>,             // 消息内容
    send_user: String,        // 发送者 uuid
    target_user: String,      // 目标接收者 uuid
    timestamp: i64,
    platform: String,         // 来源平台（PC/MOBILE）
}

/// 转发结果
struct ForwardResult {
    success: bool,
    delivered: bool,          // true=在线送达, false=不在线
    error: Option<String>,    // 失败原因
}
```

### 5.4 节点间连接池

```rust
// key = "192.168.1.20:4433" → QUIC Connection
lazy_static! {
    static ref NODE_CONNECTIONS: RwLock<HashMap<String, quinn::Connection>> =
        RwLock::new(HashMap::new());
}
```

**惰性连接**：不预配对端列表，需要转发到某个地址时才建立连接，建好后缓存复用。

## 6. 路由转发流程

### 6.1 完整时序

```
user1                    Node A :4433              Redis                 Node D :4433              user2
  │                         │                       │                      │                      │
  │── QUIC 消息 ──────────► │                       │                      │                      │
  │                         │── 查本地 HashMap ───► │                      │                      │
  │                         │  user2 不在本机       │                      │                      │
  │                         │── GET user2 ────────► │                      │                      │
  │                         │◄─ "192.168.1.20:4433"│                      │                      │
  │                         │                       │                      │                      │
  │                         │── 懒连接 Node D ──────────────────────────► │                      │
  │                         │  存入 NODE_CONNECTIONS │                      │                      │
  │                         │                       │                      │                      │
  │                         │── open_bi() 新流 ──────────────────────────► │                      │
  │                         │   send(InterNodePayload) │                   │── 写 SendStream ────►│
  │                         │   finish()              │                   │                      │
  │                         │                         │                   │◄─ 应用层确认 ────────│
  │                         │◄─ ForwardResult ──────────────────────────── │                      │
  │                         │   { delivered: true }    │                   │                      │
  │                         │                          │                   │                      │
  │                         │── 写入 WAL (chat_messages.log) │                   │                      │
  │                         │   页缓存刷盘                   │                   │                      │
  │◄─ ACK(RECALL_SUCCESS) ──│                              │                   │                      │
```

### 6.2 路由逻辑

```rust
async fn route_message(text_msg: TextQuicMsg, platform: &str) -> Result<()> {
    let recv_user = &text_msg.recv_user;

    // 1. 查本地：用户是否在本节点上
    if try_deliver_local(recv_user, &packet).await {
        return Ok(());  // 已本地投递
    }

    // 2. 查 Redis：获取用户所在节点的地址
    let addr = query_redis_route(recv_user, platform).await?;

    // 3. 跨节点转发（开新流，等 ACK）
    if let Some(target_addr) = addr {
        let result = forward_to_node(&target_addr, &payload).await?;
        // result.delivered 表示是否在线投递成功
    }

    // 4. TODO: 写入本地 WAL (chat_messages.log)
    // append_wal(&text_msg).await?;

    Ok(())
}
```

### 6.3 ACK 时序

**当前简化策略**：消息投递后**直接返回 ACK**，不等待 WAL 持久化。

```
用户发送消息
    │
    ├── 在线转发（本机/跨节点）
    │      └── 收到 ForwardResult → 继续
    │
    ├── [TODO] 写入本地 WAL (chat_messages.log)
    │
    └── 回 ACK 给发送方 (RECALL_SUCCESS)
```

> **注意**：TODO 中的 WAL 持久化实现后，ACK 时机调整为**页缓存写入完成后返回**，不等待 fsync 也不等待下游消费。

## 7. 消息持久化（TODO）

### 7.1 当前逻辑

[process_msg_service.rs](../crates/quic_service/src/msg_service/process_msg_service.rs#L211-L226) 中的 `add_user_chat_record()` 直接写 PostgreSQL。

### 7.2 目标方案：本地 WAL + 水位监控

不使用消息队列中间件，改为**本地写 Ahead Log + 异步同步**：

```
用户发送消息
    │
    ├── 在线转发（本机 / 跨节点）
    │      └── 收到 ForwardResult → 继续
    │
    ├── 写入本地 chat_messages.log（操作系统页缓存）
    │      └── 页缓存刷盘 → 返回 ACK 给用户
    │
    └── [后台独立任务] WALReader
           ├── 定时扫描 chat_messages.log
           ├── 读取水位之后的新增记录
           ├── 投递到下游（如 Kafka / 直接写 PostgreSQL）
           └── 更新水位（记录已安全消费到的位置）
```

### 7.3 三个角色

#### 角色一：消息写入器（当前路由流程内同步调用）

```
收到用户消息 → 在线转发完成后

chat_messages.log 文件格式：
┌─────────────────────────────────────────┐
│  [8字节 消息长度] [bincode 序列化消息体]  │
│  [8字节 消息长度] [bincode 序列化消息体]  │
│  ...                                     │
└─────────────────────────────────────────┘
```

写入流程：
1. `append-only` 追加到文件末尾
2. 写入完成即返回（操作系统页缓存）
3. 后台 fsync 策略保证不丢（可配置：每条刷盘 / 每 N 条 / 每 N 毫秒）

> ACK 给用户的时机：**页缓存写入完成后即可返回**，不等待 fsync 也不等待下游消费。
> 如果愿意牺牲极端故障场景（如掉电丢几毫秒的数据），甚至可以纯 write 不 fsync 就回 ACK。

#### 角色二：水位文件

```
chat_messages.watermark
┌────────────────────┐
│  184723            │  ← 已消费到的字节偏移量
└────────────────────┘
```

- 单调递增，仅 WALReader 写入
- 节点重启时，WALReader 从水位处开始继续读取

#### 角色三：WALReader（后台独立任务）

```
loop {
    1. 读取 chat_messages.watermark，得到水位偏移量
    2. 打开 chat_messages.log，seek 到水位位置
    3. 按 [长度 + 数据] 格式读取新消息
    4. 批量投递给下游消费者（MQ / 直接写 PostgreSQL）
    5. 消费成功后，更新水位文件
    6. 若没有新数据，sleep 一段时间再扫描
}
```

### 7.4 待考虑的问题（后续细化）

- **日志轮转**：`chat_messages.log` 达到阈值后切分，旧的压缩/清理（需确保水位之下的日志已无消费端依赖）
- **故障恢复**：节点崩溃重启后，WALReader 从水位处续读，确保不丢数据
- **去重**：每条消息有 `nano_id`，下游写入 PostgreSQL 时可通过唯一约束去重

### 7.5 当前阶段

**暂不实现**，路由转发完成后直接返回 ACK，预留好代码扩展点：

## 8. HTTP ↔ QUIC 服务间通信

### 8.1 现有耦合问题

当前 `api` crate 同时依赖 `http_service` 和 `quic_service`，在 [user_integrated_service.rs](../crates/api/src/service/user_integrated_service.rs#L11) 中直接调用了 `send_quic_system_msg()`：

```rust
use quic_service::msg_service::send_msg::send_quic_system_msg;

// 在 HTTP 请求处理流程中直接调用 QUIC 函数
send_quic_system_msg(target_id, NOTIFY_TYPE_MSG, json_str).await?;
```

这在单机模式下没问题，但在集群环境下 `send_quic_system_msg()` 只查本机 `GLOBAL_QUIC_SERVER_LIST`，**目标用户在另一个节点时投递会静默失败**。同时在架构上，`http_service` 和 `quic_service` 应该完全解耦，只通过**网络通信**交互。

### 8.2 通信方式

`http_service` 如果要发送消息到 QUIC 连接（如发送系统通知），应采用和节点间通信一致的**新流 + 请求/响应**模式，扮演 QUIC 客户端角色：

```
HTTP 请求处理流程
    │
    ├── 处理业务逻辑（写 DB 等）
    │
    ├── 需要发送 QUIC 消息时：
    │      ├── 作为 QUIC 客户端连接本地 QUIC 服务端 (127.0.0.1:4433)
    │      ├── 魔术字节: 0x20（标识 HTTP 服务发起的请求）
    │      ├── open_bi() 新流
    │      │    ├── send(HttpServicePayload) 
    │      │    └── finish()
    │      ├── 等 RecvStream 响应
    │      │    └── recv → HttpServiceResult
    │      └── 关闭流
    │
    └── 返回 HTTP 响应
```

与节点间通信的通道复用方式不同，HTTP 服务每次需要发送 QUIC 消息时都**重新建立连接**（因为 HTTP 服务与 QUIC 服务部署在同一台机器，`127.0.0.1` 的 QUIC 连接成本很低，且 QUIC 的 0-RTT 握手可以复用之前的连接凭证）。

### 8.3 新增请求类型：HTTP → QUIC

新增魔术字节 `0x20` 表示 "HTTP 服务请求"：

| 魔术字节 | 来源 | 说明 |
|----------|------|------|
| `0x20` | HTTP 服务 | 连到本地 QUIC 服务，发送系统通知等请求 |

新增请求/响应协议：

```rust
/// HTTP 服务 → QUIC 服务的请求体
struct HttpServicePayload {
    action: HttpAction,        // 操作类型
    body: Vec<u8>,             // 具体操作的参数（bincode 序列化）
}

enum HttpAction {
    SendSystemMsg = 1,         // 发送系统通知
    QueryOnlineUsers = 2,      // 查询某个用户是否在线（预留）
    // 后续可扩展
}

/// QUIC 服务 → HTTP 服务的响应
struct HttpServiceResult {
    success: bool,
    error: Option<String>,
    data: Option<Vec<u8>>,     // 返回数据（如查询结果）
}
```

`0x20` 请求的处理直接复用 `handle_inter_node_request()` 的逻辑，因为最终也是查 `GLOBAL_QUIC_SERVER_LIST` 投递，区别只是连接来源不同。

### 8.4 在线节点列表 API

新增 HTTP 接口：`GET /integrated/online_nodes`

返回所有注册在 Redis 中的 QUIC 服务节点信息。

**实现方式**：

### 8.5 `api` crate 的改造

`api` crate 是整个系统中**唯一**同时引用 `http_service` 和 `quic_service` 的地方。改造后 `api` crate 不再直接调用 `quic_service`，改为通过网络请求：

```rust
// 改造前（api crate）
use quic_service::msg_service::send_msg::send_quic_system_msg;
send_quic_system_msg(target_id, ...).await?;

// 改造后（api crate → http_service）
// 方式一：HTTP 服务提供一个内部 API，由 QUIC 客户端调用
// 方式二：api 层自己充当 QUIC 客户端，连接本地 QUIC 服务
// 
// 推荐方式二，因为 api crate 启动时 QUIC 服务已在同一进程运行，
// 但为了彻底解耦，api crate 应移除对 quic_service 的依赖。
```

**最终目标**：`api` crate 的 `Cargo.toml` 中移除 `quic_service` 依赖，所有 QUIC 通信走网络。

### 8.6 `handle_connection` 魔术字节扩展

更新 §4 中的魔术字节表：

| 魔术字节 | 连接类型 | 后续协议 |
|----------|---------|---------|
| `0x01` | 节点间连接 | 4 字节长度前缀 + bincode 的 `InterNodePayload` |
| `0x10` | 客户端连接 | 剩余字节作为 JSON 解析（现有 `FirstQuicMsg` 逻辑） |
| `0x20` | HTTP 服务请求 | 4 字节长度前缀 + bincode 的 `HttpServicePayload` |

`0x20` 的处理和 `0x01` 基本相同，也是 `read_to_end` → 反序列化 → 执行 → 回响应。

## 9. 启动模式拆分

### 9.1 现状

当前只有一个二进制入口 `src/main.rs`，它在同一进程中依次启动：

```
src/main.rs
  ├── run_udp_server()         ← quic_service::p2p_service
  ├── start_server()           ← quic_service::init_server (QUIC 监听)
  └── init_server::start_server() ← api::init_server (HTTP + 全量初始化)
```

`api::init_server::start_server()` 承担了全部初始化工作：
1. 加载并解析 `app_config.toml` → 填充 `GLOBAL_CONFIG` DashMap
2. 初始化 PostgreSQL 连接池 → 填充 `RBATIS_DATABASE`
3. 初始化 Redis 连接池 → 填充 `REDIS_CLIENT`
4. 初始化 S3 客户端
5. 启动 HTTP 服务器

`quic_service::init_server::start_server()` 只做一件事：
1. 读取 `app_config.toml` 解析 `quic_server.address`
2. 启动 QUIC 监听

### 9.2 问题

集群模式下，`api`（HTTP 服务）和 `quic_service`（QUIC 网关）应当**各自独立部署**，各自有独立的 `main.rs`：

| 入口 | 职责 | 部署方式 |
|------|------|---------|
| `src/main.rs` | 单机模式，两者合一 | 全量部署 |
| `api/src/main.rs` | 集群模式，仅 HTTP 服务 | 每个节点部署 |
| `quic_service/src/main.rs` | 集群模式，仅 QUIC 网关 | 每个节点部署 |

三个入口都加载 `app_config.toml`，会涉及全局变量的重复初始化。

### 9.3 全局变量一览

当前系统中需要在启动时初始化的全局变量：

| 全局变量 | 类型 | 初始化位置 | 能否重复初始化 |
|----------|------|-----------|--------------|
| `GLOBAL_CONFIG` | DashMap | `init_global_config!` 宏 | ⚠️ 重复调用会覆盖已有值 |
| `REDIS_CLIENT` | `RwLock<Option<Pool>>` | `REDIS_CLIENT.write()` = Some(pool) | ⚠️ 重复调用会覆盖 |
| `RBATIS_DATABASE` | `RwLock<Option<RBatis>>` | `RBATIS_DATABASE.write()` = Some(rb) | ⚠️ 重复调用会覆盖 |
| `NODE_ADDR` | `OnceLock<String>` | `NODE_ADDR.set(addr)` | ❌ OnceLock 只允许 set 一次 |

### 9.4 初始化职责拆分

每个入口需要初始化的全局变量不同：

| 全局变量 | 单机 `src/main.rs` | API `api/main.rs` | QUIC `quic_service/main.rs` |
|----------|:---:|:---:|:---:|
| `GLOBAL_CONFIG` | ✅ | ✅ | ✅ |
| `REDIS_CLIENT` | ✅ | ✅ | ✅ |
| `RBATIS_DATABASE` | ✅ | ✅ | ❌ 不需要 |
| `NODE_ADDR` | ✅ | ❌ 不需要 | ✅ |
| S3 客户端 | ✅ | ✅ | ❌ 不需要 |
| HTTP 服务器 | ✅ | ✅ | ❌ 不需要 |
| QUIC 服务器 | ✅ | ❌ 不需要 | ✅ |
| UDP P2P 服务 | ✅ | ❌ 不需要 | ✅ |

### 9.5 拆分策略

#### 方案：提取共享初始化函数，加前置保护

在 `entity` crate 中新增 `bootstrap` 模块，提供带保护的初始化函数：

```rust
// entity/src/bootstrap.rs

use crate::config_manager;
use crate::{REDIS_CLIENT, RBATIS_DATABASE};

/// 加载配置到 GLOBAL_CONFIG
/// 幂等：重复调用更新配置值，但不影响已初始化的连接池等
pub fn load_config(path: &str) -> HashMap<String, toml::Value> {
    // ... 读取文件、解析 toml、init_global_config! ...
}

/// 初始化 Redis 连接池（带保护：已初始化则跳过）
pub async fn init_redis(url: &str) -> Pool {
    let guard = REDIS_CLIENT.read().await;
    if guard.is_some() {
        // 已初始化，直接返回
        return guard.as_ref().unwrap().clone();
    }
    drop(guard);
    // 实际初始化逻辑...
}

/// 初始化数据库连接池（带保护）
pub async fn init_database(url: &str) -> RBatis {
    let guard = RBATIS_DATABASE.read().await;
    if guard.is_some() {
        return guard.as_ref().unwrap().clone();
    }
    drop(guard);
    // 实际初始化逻辑...
}
```

#### 各入口的启动序列

**单机模式 `src/main.rs`：**

```rust
#[actix_web::main]
async fn main() {
    // 1. 加载配置（一次）
    entity::bootstrap::load_config("./config/app_config.toml");
    
    // 2. 初始化基础设施（一次，带幂等保护）
    let redis_url = read_global_config!("redis", "url");
    entity::bootstrap::init_redis(&redis_url).await;
    
    let db_url = read_global_config!("database", "url");
    entity::bootstrap::init_database(&db_url).await;
    
    // 3. 启动所有服务
    run_udp_server().await;
    quic_service::init_server::start_server().await;  // 只需启动 QUIC 监听
    api::init_server::start_server().await;             // 只需启动 HTTP 监听
}
```

**集群模式 — QUIC 网关 `quic_service/src/main.rs`：**

```rust
#[tokio::main]
async fn main() {
    // 1. 加载配置
    entity::bootstrap::load_config("./config/app_config.toml");
    
    // 2. 初始化 Redis（路由查询用）
    let redis_url = read_global_config!("redis", "url");
    entity::bootstrap::init_redis(&redis_url).await;
    
    // 3. 初始化节点标识
    let node_addr = read_global_config!("quic_server", "node_address");
    config_str::NODE_ADDR.set(node_addr).unwrap();
    
    // 4. 启动 QUIC 服务 + UDP P2P
    run_udp_server().await;
    quic_service::init_server::start_server().await;
}
```

**集群模式 — API HTTP 服务 `api/src/main.rs`：**

```rust
#[actix_web::main]
async fn main() {
    // 1. 加载配置
    entity::bootstrap::load_config("./config/app_config.toml");
    
    // 2. 初始化 DB、Redis、S3
    let db_url = read_global_config!("database", "url");
    entity::bootstrap::init_database(&db_url).await;
    
    let redis_url = read_global_config!("redis", "url");
    entity::bootstrap::init_redis(&redis_url).await;
    
    // 3. 启动 HTTP 服务
    api::init_server::start_server().await;
}
```

### 9.7 单机兼容性保障

`src/main.rs`（单机模式）的启动流程改为：

```
1. entity::bootstrap::load_config()
2. entity::bootstrap::init_redis()    ← 幂等
3. entity::bootstrap::init_database() ← 幂等
4. run_udp_server()
5. quic_service::init_server::start_server()    ← 跳过内部 init，仅启动 QUIC
6. api::init_server::start_server()               ← 跳过内部 init，仅启动 HTTP
```

`quic_service::init_server::start_server()` 和 `api::init_server::start_server()` 需要改造成**仅启动服务，不重复初始化**。具体做法：

```rust
// quic_service/src/init_server.rs — 改造后
pub async fn start_server() -> anyhow::Result<()> {
    // 只启动 QUIC 监听，不加载配置、不初始化全局变量
    let addr = read_global_config!("quic_server", "address");
    quic_server::init_server(addr.parse()?);
    Ok(())
}

// 新增：全量启动函数（保留旧逻辑，但仅被 quic_service/main.rs 调用）
pub async fn start_server_standalone() -> anyhow::Result<()> {
    // 加载配置、初始化 Redis、设置 NODE_ADDR、启动 QUIC
    // ...
    start_server().await
}
```
