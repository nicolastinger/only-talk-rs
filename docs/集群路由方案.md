# IM 集群路由方案

## 1. 目标

将现有的单机 IM 系统升级为**多节点集群**，实现：

- 外网 QUIC 节点**只负责客户端连接管理与本机投递**，不参与节点间通信
- 内网 QUIC 节点作为**集群统一路由层**，承担所有跨节点转发（HTTP 通知 + QUIC 消息转发）
- 节点间通信采用 **新流 + 请求/响应** 模式，利用现有 `InternalQuicRequest/Response` 协议
- **确定性用户分配**：通过 `hash(uuid) % server_count` 将用户固定分配到指定外网 QUIC 节点
- 节点自动发现，通过 Redis 注册，有序号索引

> **三个关键设计**：
> 1. 外网 QUIC 只做"本机客户端 ↔ 本机内网 QUIC"的桥接
> 2. 内网 QUIC 是集群的**统一路由层**，通过 `hash(uuid) % N` 定位目标节点
> 3. HTTP API 返回给客户端的只是**分配给该用户的唯一节点**，不是全部节点列表

## 3. 确定性分配：`hash(uuid) % server_count`

### 3.1 原则

- 每个用户通过 UUID 的 hash 取模，被**确定性**分配到一个外网 QUIC 节点
- 同一用户的所有设备（PC / Mobile）连接**同一个**外网 QUIC 节点
- 路由转发时，直接 `hash(target_uuid) % server_count` 得到目标节点序号，无需遍历

### 3.2 Hash 函数

```rust
use std::collections::hash_map::DefaultHasher;
use std::hash::{Hash, Hasher};

fn compute_server_index(uuid: &str, server_count: u32) -> u32 {
    let mut hasher = DefaultHasher::new();
    uuid.hash(&mut hasher);
    let hash = hasher.finish();
    (hash as u32) % server_count
}
```

> 使用标准库 `DefaultHasher`，足够均匀且无外部依赖。

### 3.3 用户连接流程

```
1. 客户端请求 HTTP API: GET /quic_servers
     请求头携带 JWT token → 解析出 uuid
     
2. HTTP API 计算: index = hash(uuid) % server_count
     查 Redis: QUIC:SERVER:EXTERNAL:{index}
     返回: { "index": 0, "address": "192.168.1.10:4433" }

3. 客户端使用返回的 address 连接外网 QUIC

4. 外网 QUIC 鉴权后，注册到 Redis:
     PC:QUIC:SERVER:user1:TEXT → "0" (只存序号，不存地址)
```

## 4. 连接类型：双端口各司其职

### 4.1 端口职责

| 端口 | 服务 | 连接方 | 职责 |
|------|------|--------|------|
| `:4433` | 外网 QUIC | 客户端 App | 1. 客户端鉴权与连接管理<br>2. 接收/投递本机客户端消息<br>3. 本机找不到目标时 → 发给本机内网 QUIC |
| `:4434` | 内网 QUIC | HTTP API +<br>外网 QUIC | 1. 接收所有转发请求<br>2. `hash(target_uuid) % N` 定位目标节点<br>3. 本机 → 直接投递；其他节点 → 转发 |

### 4.2 外网 QUIC 不变的部分

外网 QUIC 的现有逻辑**绝大部分无需改动**：

- `handle_connection()` 保持 JSON `FirstQuicMsg` 鉴权流程
- bidi stream 接收循环 + uni stream 接收循环不变
- 粘包协议（`HeadMsg` + `TextQuicMsg`）不变
- `connections` DashMap 结构不变

### 4.3 客户端改动

客户端 App 连接外网 `:4433` 的协议完全不变。唯一变化：不再从 HTTP API 拿"服务器列表"，而是拿"分配给自己的唯一节点地址"。

## 5. 配置 + 动态 server_count

### 5.1 配置文件（静态部分）

每个节点只需配置自己的序号和地址，`server_count` 不从配置读取：

```toml
[cluster]
server_index = 0                     # 本节点序号，集群内唯一，不可重复

[quic_server]
address = "0.0.0.0:4433"
server_name = "127.0.0.1:4433"       # TLS SNI（不变）
node_address = "192.168.1.10:4433"   # 本节点供客户端连接的地址

[internal_quic_server]
address = "0.0.0.0:4434"
server_name = "INTERNAL_SERVER_1"
node_address = "192.168.1.10:4434"   # 本节点内网 QUIC 地址
```

### 5.2 `server_count` 动态获取

节点总数**不写在配置文件**中，而是从 Redis 同步。每个节点 3s 轮询一次。

**Redis 存储**：

| Key | Value | 写入方 | 写入时机 |
|-----|-------|--------|---------|
| `CLUSTER:SERVER_COUNT` | `3` | 每个节点 | 启动时执行 `INCR`，关闭时执行 `DECR` |

**节点启动**：
```rust
let count = redis_incr("CLUSTER:SERVER_COUNT").await?;
// count = 1, 2, 3... 即当前集群节点总数
```

**节点关闭**（优雅退出）：
```rust
redis_decr("CLUSTER:SERVER_COUNT").await?;
```

### 5.3 后台同步任务

```rust
use std::sync::atomic::{AtomicU32, Ordering};
use std::sync::Arc;

/// 全局 server_count，lock-free 读取
pub static SERVER_COUNT: LazyLock<Arc<AtomicU32>> =
    LazyLock::new(|| Arc::new(AtomicU32::new(1)));

/// 启动后台同步任务（每个节点启动时调用一次）
pub fn start_server_count_sync(redis_pool: Pool) {
    tokio::spawn(async move {
        let mut interval = tokio::time::interval(Duration::from_secs(3));
        loop {
            interval.tick().await;
            match redis_get::<u32>("CLUSTER:SERVER_COUNT").await {
                Ok(count) if count > 0 => {
                    let old = SERVER_COUNT.swap(count, Ordering::Relaxed);
                    if old != count {
                        info!("server_count 更新: {} → {}", old, count);
                    }
                }
                Err(e) => {
                    warn!("同步 server_count 失败: {}", e);
                }
            }
        }
    });
}

/// 任意位置读取当前 server_count
fn get_server_count() -> u32 {
    SERVER_COUNT.load(Ordering::Relaxed)
}
```

> 使用 `AtomicU32` 而非 `Arc<RwLock<u32>>`，读取零锁开销。路由函数是热点路径。

## 6. 路由转发流程

### 6.1 路由策略：hash 取模优先 + Redis 兜底

```
路由决策树：

  1. hash(uuid) % server_count → 首选节点序号
          │
          ▼
  2. 首选节点 == 本机？
     ├── 是 → 查本地 connections
     │        ├── 在线 → 投递 ✅
     │        └── 不在线 → 进入第 3 步（Redis 兜底）
     │
     └── 否 → 转发到首选节点的内网 QUIC
               └── 对端同样走第 2 步
                    ├── 在线 → 投递 ✅
                    └── 不在线 → 返回 user_offline
                                   │
                                   ▼
  3. Redis 兜底：查 PC:QUIC:SERVER:{uuid}:TEXT
          │
          ├── 有值（用户在另一个节点在线）
          │     → 转发到实际节点的内网 QUIC
          │     → 投递 (ttl 耗尽则放弃)
          │
          └── 无值（用户确实不在线）
                → 返回 user_offline
```

> **为什么需要 Redis 兜底**：在节点扩缩容或节点故障时，用户可能连接到非 hash 分配的节点（客户端重新获取地址后连接到了其他节点）。hash 取模是首选路由，Redis 兜底覆盖异常情况。

### 6.2 核心路由函数

```rust
/// 根据目标用户 UUID 计算首选节点序号（从 AtomicU32 读 server_count）
fn compute_preferred_index(uuid: &str) -> u32 {
    let sc = SERVER_COUNT.load(Ordering::Relaxed);
    if sc <= 1 { return 0; }
    let mut hasher = DefaultHasher::new();
    uuid.hash(&mut hasher);
    (hasher.finish() as u32) % sc
}

/// 查 Redis 获取某序号节点的内网 QUIC 地址
async fn get_internal_addr_by_index(index: u32) -> Result<SocketAddr> {
    let key = format!("INTERNAL:QUIC:SERVER:{}", index);
    let addr_str: String = redis_get(&key).await?;
    addr_str.parse()
}

/// Redis 兜底：查用户实际在哪个节点在线
async fn get_actual_node_index(uuid: &str, platform: &str) -> Result<Option<u32>> {
    let key = format!("{}:QUIC:SERVER:{}:{}", platform, uuid, "TEXT");
    let index_str: Option<String> = redis_get_opt(&key).await?;
    Ok(index_str.and_then(|s| s.parse().ok()))
}
```

### 6.3 内网 QUIC 路由（核心 — 两阶段）

```rust
async fn route_request(
    request: &InternalQuicRequest,
    connections: &Arc<DashMap<String, QuicConnection>>,
    server_index: u32,
) -> Result<InternalQuicResponse> {
    let preferred_index = request.preferred_index;

    // ===== 阶段一：hash 取模路由 =====

    // 首选节点 == 本机 → 尝试本机投递
    if preferred_index == server_index {
        if let Some(resp) = try_deliver_local(request, connections).await? {
            return Ok(resp);  // 在线，投递成功
        }
        // 本机未找到 → 不走 else 分支，直接进入 Redis 兜底
    } else {
        // 首选节点 ≠ 本机 → 先转发到首选节点试试
        if request.ttl > 0 {
            let target_addr = get_internal_addr_by_index(preferred_index).await?;
            let mut forward_req = request.clone();
            forward_req.ttl -= 1;
            let resp = forward_to_remote(&target_addr, &forward_req).await?;
            if resp.delivered == Some(true) {
                return Ok(resp);  // 首选节点投递成功
            }
        }
    }
    // 首选路由未命中 → 进入 Redis 兜底

    // ===== 阶段二：Redis 兜底 =====

    if request.ttl == 0 {
        return Ok(InternalQuicResponse::user_offline());
    }

    // 查用户实际在哪个节点在线
    let actual_index = get_actual_node_index(&request.target_user, &request.platform).await?;

    match actual_index {
        Some(idx) if idx == server_index => {
            // 实际就在本机（之前阶段一没找到？再试一次）
            try_deliver_local(request, connections).await
                .map(|r| r.unwrap_or_else(InternalQuicResponse::user_offline))
        }
        Some(idx) => {
            // 转发到实际节点
            let target_addr = get_internal_addr_by_index(idx).await?;
            let mut forward_req = request.clone();
            forward_req.ttl -= 1;
            forward_to_remote(&target_addr, &forward_req).await
        }
        None => {
            // 确实不在线
            Ok(InternalQuicResponse::user_offline())
        }
    }
}
```

### 6.4 外网 QUIC → 内网 QUIC（本机）

外网 QUIC 的 `send_msg_to_user()` 改造：

```rust
async fn send_msg_to_user(
    text_msg: TextQuicMsg,
    platform: &str,
    connections: &Arc<DashMap<String, QuicConnection>>,
    server_index: u32,
) -> anyhow::Result<()> {
    let res = generate_text_msg_with_time(/* ... */)?;
    let preferred_index = compute_preferred_index(&text_msg.recv_user);

    for target_platform in [PC_PLATFORM, MOBILE_PLATFORM] {
        let key = format_key(target_platform, &text_msg.recv_user);

        // 1. 首选节点 == 本机 → 查本地 connections 直接投递
        if preferred_index == server_index {
            if let Some(entry) = connections.get(&key) {
                let mut send = entry.conn.open_uni().await?;
                send.write_all(&res).await?;
                send.finish().await?;
                continue;
            }
            // 本机也未找到 → 不直接放弃，交给内网 QUIC 走两阶段路由
        }

        // 2. 发给本机内网 QUIC（它会走 hash 优先 + Redis 兜底）
        let request = InternalQuicRequest {
            msg_type: text_msg.text_type,
            payload: String::from_utf8_lossy(&res).to_string(),
            target_user: text_msg.recv_user.clone(),
            preferred_index,
            platform: target_platform.to_string(),
            source: RequestSource::QuicExternal,
            ttl: 3,
        };
        let internal_addr = get_local_internal_addr();
        send_internal_quic_msg(internal_addr, request).await?;
    }

    Ok(())
}
```

> `server_count` 由 `compute_preferred_index()` 内部从 `SERVER_COUNT` 原子变量读取，不需要作为参数传递。`server_index` 从 `ChatNodeConfig` 传入但不再依赖 `server_count`。


### 6.7 HTTP API → 内网 QUIC

HTTP API 的好友通知等场景，走同样的两阶段路由：

```
HTTP API → hash(target_uuid) % N → 构造 InternalQuicRequest { preferred_index: X, ... }
         → 发给本机内网 QUIC
              ├── 阶段一：X == 本机序号 → 本机投递
              │           X != 本机序号 → 转发到首选节点
              └── 阶段二（未命中）：Redis 兜底 → 查实际在线节点 → 再次转发
```

## 7. Redis 注册

### 7.1 Key 设计

| Key 模式 | Value | 用途 | 写入方 |
|----------|-------|------|--------|
| `CLUSTER:SERVER_COUNT` | `3` | **节点总数**：INCR/DECR | 每个节点启动/关闭时 |
| `QUIC:SERVER:EXTERNAL:{index}` | `{node_address}` | **节点地址映射**：序号 → 外网地址 | 外网 QUIC 启动时 |
| `INTERNAL:QUIC:SERVER:{index}` | `{node_address}` | **内网地址映射**：序号 → 内网地址 | 内网 QUIC 启动时 |
| `PC:QUIC:SERVER:{uuid}:TEXT` | `{index}` | **用户路由**：用户 → 所在节点序号 | 外网 QUIC `set_conn_info()` |

### 7.2 注册时机

| 事件 | 操作 |
|------|------|
| QUIC 节点启动 | `INCR CLUSTER:SERVER_COUNT`；外网注册 `QUIC:SERVER:EXTERNAL:{index}`；内网注册 `INTERNAL:QUIC:SERVER:{index}`；启动 `start_server_count_sync()` |
| 用户上线 | `set_conn_info()` 写 `PC:QUIC:SERVER:{uuid}:TEXT` → `{index}` |
| 用户下线 | `end_server()` 删 `PC:QUIC:SERVER:{uuid}:TEXT` |
| QUIC 节点关闭 | `DECR CLUSTER:SERVER_COUNT`；删 `QUIC:SERVER:EXTERNAL:{index}` 和 `INTERNAL:QUIC:SERVER:{index}` |

### 7.3 合并注册逻辑

- 外网 QUIC 注册：节点地址 + 用户路由
- 内网 QUIC 注册：内网地址（供跨节点转发用）
- server_count 由 INCR/DECR 维护
- **不重复注册**，各负其责

## 8. HTTP API：`GET /quic_servers`

### 8.1 功能

客户端获取**分配给自己的唯一外网 QUIC 节点地址**。

### 8.2 请求

```
GET /integrated/quic_servers
Header: Authorization: Bearer {jwt}
```

### 8.3 处理逻辑

```rust
pub async fn get_quic_server_for_user(uuid: &str) -> Result<QuicServerInfo> {
    // 1. hash 取模定位节点序号（server_count 从 SERVER_COUNT 原子变量读取）
    let index = compute_preferred_index(uuid);

    // 2. 查 Redis 获取该节点的外网地址
    let key = format!("QUIC:SERVER:EXTERNAL:{}", index);
    let address: String = redis_get(&key).await?;

    Ok(QuicServerInfo { index, address })
}
```

### 8.4 响应

```json
{
    "code": 0,
    "data": {
        "index": 0,
        "address": "192.168.1.10:4433"
    }
}
```

> **关键变化**：旧方案返回全部服务器列表，新方案只返回分配给该用户的**一个**节点。

## 14. 方案对比总结

| 维度 | 初版（魔术字节） | 二版（内网路由） | 三版（hash 取模） |
|------|:---:|:---:|:---:|
| 外网 QUIC 职责 | 客户端 + 节点间 | 仅客户端 | 仅客户端 |
| 路由方式 | 魔术字节分发 | 查 Redis 遍历 | **hash 取模一次定位** |
| 用户分配 | 随机（先到先得） | 随机 | **确定性分配** |
| HTTP 返回 | 全部服务器列表 | 全部服务器列表 | **只返回分配的一个** |
| 客户端感知 | 需发魔术字节 | 无变化 | **无变化** |
| 节点间协议 | 新 inter_node | 复用内网 QUIC | 复用内网 QUIC |
| 新增模块 | `inter_node/` | `internal_router.rs` | `internal_router.rs` + `hash_router.rs` |
