# IM 集群路由方案

## 1. 目标

将现有的单机 IM 系统升级为**多节点集群**，实现：

- QUIC 网关节点**无状态**，只做消息转发 + ACK
- 消息持久化交由消息队列异步处理（暂不实现，留 TODO）
- 节点间通信采用 **新流 + 请求/响应** 模式，避免粘包处理
- 节点自动发现，无需手动配置对端列表

---

## 3. 节点标识

### 3.1 现状

`server_name` 已支持从配置文件读取（[ChatNodeConfig::from_toml_str()](../crates/quic_service/src/config.rs#L48)）：

```toml
# config/app_config.toml — 当前已有
[quic_server]
server_name="127.0.0.1:4433"
```

`config_str.rs` 中仍保留 `SERVER_NAME = "SERVER_1"` 常量（[行12](../crates/entity/src/config_str.rs#L12)），但实际运行时 `set_conn_info()` 使用的是 `config.server_name`（[quic_server.rs:256](../crates/quic_service/src/quic_server.rs#L256)），即 TOML 配置的值。

**问题**：`server_name` 既被用作 TLS SNI 名称，又被用作 Redis 注册的节点标识，语义混杂。集群模式下需要一个**明确的可路由地址**作为节点唯一标识。

### 3.2 目标：新增 `node_address` 字段

在 `app_config.toml` 的 `[quic_server]` 下新增 `node_address` 字段：

```toml
[quic_server]
address="0.0.0.0:4433"              # QUIC 监听地址（保持不变）
server_name="127.0.0.1:4433"        # TLS SNI 名称（保持不变）
node_address="192.168.1.10:4433"    # 新增：本节点供其他节点连接的目标地址
```

每个节点部署时各自配置 `node_address`：

| 节点 | `node_address` 值 |
|------|-------------------|
| Node A | `192.168.1.10:4433` |
| Node B | `192.168.1.11:4433` |
| Node D | `192.168.1.20:4433` |

> `address` 是服务端 `bind()` 监听地址（通常 `0.0.0.0:4433`）；
> `server_name` 是 TLS 证书校验用的 SNI 名称（保持 `127.0.0.1:4433` 不变）；
> `node_address` 是**对端可达的 `ip:port`**，用于节点间建连和 Redis 注册。

### 3.3 `ChatNodeConfig` 扩展

[config.rs](../crates/quic_service/src/config.rs) 中 `ChatNodeConfig` 结构体新增字段：

|--------|--------|

## 4. 连接类型区分

### 4.1 现状：双端口模型

当前代码已有两种 QUIC 服务端口：

| 端口 | 服务 | 文件 |
|------|------|------|
| `:4433` | 外网 QUIC（客户端连接） | [quic_server.rs](../crates/quic_service/src/quic_server.rs) |
| `:4434` | 内网 QUIC（API → QUIC） | [internal_quic_server.rs](../crates/quic_service/src/internal_quic_server.rs) |

- 外网 QUIC 通过 JSON 格式的 `FirstQuicMsg` 鉴权，数据帧为自定义粘包协议
- 内网 QUIC 通过 JSON 格式的 `InternalQuicRequest` / `InternalQuicResponse` 交互，已实现请求-响应模式

### 4.2 目标：单端口 + 魔术字节分发

集群模式下，**外网端口 `:4433` 需要同时接受客户端连接和其他节点的连接**。方案是在第一条流的首字节加入魔术字节区分：

| 魔术字节 | 连接类型 | 后续协议 |
|----------|---------|---------|
| `0x01` | 节点间连接 | 4 字节长度前缀 + bincode 序列化的 `InterNodePayload` |
| `0x10` | 客户端连接 | 剩余字节作为 JSON 解析（现有 `FirstQuicMsg` 逻辑） |

> 内网服务请求（`0x20`）仍走独立的 `:4434` 端口，不合并到外网端口。

### 4.3 目标：连接分发逻辑

```rust
// quic_server.rs — handle_connection() 改造

async fn handle_connection(conn: Connection, ...) -> Result<()> {
    let (mut send, mut recv) = conn.accept_bi().await?;
    let mut magic = [0u8; 1];
    recv.read_exact(&mut magic).await?;

    match magic[0] {
        0x01 => {
            handle_inter_node_conn(conn, connections).await?;
        }
        0x10 => {
            handle_client_conn(send, recv, conn, address, connections, config).await?;
        }
        _ => {
            send.finish().await?;
        }
    }
}
```

### 4.4 内网 QUIC 保持不变

内网 QUIC（`:4434`）不受集群改造影响，继续独立承担"API 层 → QUIC 层"的桥接功能：

- **服务端**：`internal_quic_server.rs` 保持自签名证书 + JSON 请求/响应协议
- **客户端**：`internal_quic_client.rs` 保持跳过证书校验的连接模式
- **职责**：HTTP API 层发送系统通知、好友请求等消息时，通过内网 QUIC 转发到 QUIC 网关

唯一需要的改动是 §8.3 所述：`handle_internal_request()` 在集群模式下也要走集群路由，而非仅查本机。

### 4.5 客户端改动

客户端连接时，第一条流的首字节发送 `0x10`，后续紧跟现有的 JSON 初始化消息 `FirstQuicMsg`。

## 5. 节点间通信协议

### 5.1 设计原则

- **新流 + 请求/响应**：每个转发请求开一条 Quinn `open_bi()`，发送完毕调用 `finish()` 关闭发送端，然后阻塞读取 `RecvStream` 获取响应
- **无粘包**：每个流只包含一个请求 + 一个响应，QUIC 流天然隔离
- **超时控制**：每个转发请求设置 10 秒超时

### 5.2 数据协议

```
请求方向（发送方 → 接收方）:
┌──────────┬──────────────┬──────────────────────────────┐
│  0x01    │  4 字节长度   │    bincode 序列化的           │
│ (魔术字节)│  (大端 u32)  │    InterNodePayload           │
└──────────┴──────────────┴──────────────────────────────┘

响应方向（接收方 → 发送方）:
┌──────────────────────────────────────┐
│   bincode 序列化的 ForwardResult     │
│   (读至 stream 关闭, read_to_end)    │
└──────────────────────────────────────┘
```

> 首字节 `0x01` 由 §4 的 `handle_connection()` 层消费；节点间连接建立后的每条新流不再重复魔术字节，直接 `长度 + payload`。

### 5.3 数据结构

```rust
/// 节点间转发请求
struct InterNodePayload {
    msg_id: String,           // 消息 nano_id
    text_type: u16,           // 消息类型（文本/图片/文件...）
    raw: Vec<u8>,             // 消息内容（已序列化的 TextQuicMsg.body）
    send_user: String,        // 发送者 uuid
    target_user: String,      // 目标接收者 uuid
    timestamp: i64,
    platform: String,         // 来源平台（PC/MOBILE）
}

/// 转发结果
struct ForwardResult {
    success: bool,
    delivered: bool,          // true=在线送达, false=不在线
    error: Option<String>,    // 失败原因
}
```

### 5.4 节点间连接池

```rust
// key = "192.168.1.20:4433" → QUIC Connection
use dashmap::DashMap;
use std::sync::LazyLock;

static NODE_CONNECTIONS: LazyLock<DashMap<String, quinn::Connection>> =
    LazyLock::new(|| DashMap::new());
```

**惰性连接**：不预配对端列表，需要转发到某个地址时才建立连接，建好后缓存复用。

## 6. 路由转发流程

### 6.1 现有本机投递逻辑

当前 `send_msg_to_user()`（[process_msg_service.rs:78](../crates/quic_service/src/msg_service/process_msg_service.rs#L78)）：

1. 查 `connections` DashMap（本机客户端映射）
2. 找到 → `open_uni()` 直接写入，完成
3. 找不到 → `warn!("当前用户不在线")`，静默跳过

集群化后需要增加：本机找不到 → 查 Redis → 跨节点转发。

### 6.2 目标：完整时序

```
user1                    Node A :4433              Redis                 Node D :4433              user2
  │                         │                       │                      │                      │
  │── QUIC 消息 ──────────► │                       │                      │                      │
  │                         │── 查本地 connections ──► │                      │                      │
  │                         │  user2 不在本机       │                      │                      │
  │                         │── GET *:user2:* ────► │                      │                      │
  │                         │◄─ "192.168.1.20:4433"│                      │                      │
  │                         │                       │                      │                      │
  │                         │── 懒连接 Node D ──────────────────────────► │                      │
  │                         │  存入 NODE_CONNECTIONS  │                      │                      │
  │                         │                       │                      │                      │
  │                         │── open_bi() 新流 ──────────────────────────► │                      │
  │                         │   send(InterNodePayload) │                   │── 写 SendStream ────►│
  │                         │   finish()              │                   │                      │
  │                         │                         │                   │◄─ 应用层确认 ────────│
  │                         │◄─ ForwardResult ──────────────────────────── │                      │
  │                         │   { delivered: true }    │                   │                      │
  │                         │                          │                   │                      │
  │◄─ ACK(RECALL_SUCCESS) ──│                              │                   │                      │
```

### 6.3 改造后的路由逻辑

```rust
// process_msg_service.rs — send_msg_to_user() 改造

async fn send_msg_to_user(
    text_msg: TextQuicMsg,
    platform: &str,
    connections: &Arc<DashMap<String, QuicConnection>>,
) -> anyhow::Result<()> {
    let recv_user = &text_msg.recv_user;
    let res = generate_text_msg_with_time(
        text_msg.nano_id, text_msg.text_type, text_msg.raw,
        text_msg.recv_user, text_msg.send_user, text_msg.timestamp,
    )?;

    for target_platform in [PC_PLATFORM, MOBILE_PLATFORM] {
        let key = format!("{}:QUIC:SERVER:{}:TEXT", target_platform, recv_user).to_uppercase();

        // 1. 查本地
        if let Some(entry) = connections.get(&key) {
            let mut send = entry.conn.open_uni().await?;
            send.write_all(&res).await?;
            send.finish().await?;
            continue;
        }

        // 2. 查 Redis → 跨节点转发
        let node_addr = query_redis_route(recv_user, target_platform).await?;
        if let Some(addr) = node_addr {
            let payload = InterNodePayload { /* ... */ };
            let result = forward_to_node(&addr, &payload).await?;
            if !result.delivered {
                warn!("用户 {} 不在线（节点 {}）", recv_user, addr);
            }
        } else {
            warn!("用户 {} 未在任何节点在线", recv_user);
        }
    }

    // 回显给发送者的其他平台（保持不变）
    let mirror_platform = if platform == PC_PLATFORM { MOBILE_PLATFORM } else { PC_PLATFORM };
    // ... 现有回显逻辑 ...

    Ok(())
}
```

### 6.4 ACK 时机

**当前策略**：本地投递后立即返回 ACK。

```
用户发送消息
    │
    ├── 本机投递 / 跨节点转发
    │      └── 收到 ForwardResult → 继续
    │
    └── 回 ACK 给发送方 (RECALL_SUCCESS)
```

> 后续引入 WAL 持久化后，ACK 在页缓存写入完成后返回，不等待 fsync。

## 7. 消息持久化（TODO）

### 7.1 当前逻辑

[process_msg_service.rs](../crates/quic_service/src/msg_service/process_msg_service.rs#L67-L72) 中 `process_text_msg()` 通过 `tokio::spawn` 异步调用 `add_user_chat_record()` 直接写 PostgreSQL。

### 7.2 目标方案：本地 WAL + 水位监控

不使用消息队列中间件，改为**本地 Write-Ahead Log + 异步同步**：

```
用户发送消息
    │
    ├── 在线转发（本机 / 跨节点）
    │      └── 收到 ForwardResult → 继续
    │
    ├── 写入本地 chat_messages.log（操作系统页缓存）
    │      └── 页缓存写入 → 返回 ACK 给用户
    │
    └── [后台独立任务] WALReader
           ├── 定时扫描 chat_messages.log
           ├── 读取水位之后的新增记录
           ├── 投递到下游（Kafka / 直接写 PostgreSQL）
           └── 更新水位（记录已安全消费到的偏移量）
```

### 7.3 三个角色

#### 角色一：消息写入器（路由流程内同步调用）

```
chat_messages.log 文件格式：
┌─────────────────────────────────────────┐
│  [8字节 消息长度] [bincode 序列化消息体]  │
│  [8字节 消息长度] [bincode 序列化消息体]  │
│  ...                                     │
└─────────────────────────────────────────┘
```

写入流程：
1. `append-only` 追加到文件末尾
2. 写入完成即返回（操作系统页缓存）
3. 后台 fsync 策略可配置（每条刷盘 / 每 N 条 / 每 N 毫秒）

#### 角色二：水位文件

```
chat_messages.watermark
┌────────────────────┐
│  184723            │  ← 已消费到的字节偏移量
└────────────────────┘
```

- 单调递增，仅 WALReader 写入
- 节点重启时，WALReader 从水位处开始继续读取

#### 角色三：WALReader（后台独立任务）

```
loop {
    1. 读取 chat_messages.watermark，得到水位偏移量
    2. 打开 chat_messages.log，seek 到水位位置
    3. 按 [长度 + 数据] 格式读取新消息
    4. 批量投递给下游消费者（MQ / 直接写 PostgreSQL）
    5. 消费成功后，更新水位文件
    6. 若没有新数据，sleep 一段时间再扫描
}
```

### 7.4 待考虑问题

- **日志轮转**：`chat_messages.log` 达到阈值后切分，水位之下的日志可压缩/清理
- **故障恢复**：节点崩溃重启后，WALReader 从水位续读
- **去重**：每条消息有 `nano_id`，下游 PostgreSQL 通过唯一约束去重

### 7.5 当前阶段

**暂不实现**。路由转发完成后直接返回 ACK，预留代码扩展点。

### 8.2 问题

内网 QUIC 的 `handle_internal_request()` 调用 `send_quic_system_msg()`（[internal_quic_server.rs:73](../crates/quic_service/src/internal_quic_server.rs#L73)），后者只查本机 `connections`：

```rust
// send_msg.rs — send_quic_system_msg()

if connections.get(&user_key).is_none() {
    // 用户不在本机 → 静默跳过
    return Ok(());
}
```

**集群模式下，目标用户可能在另一个节点，投递会静默失败。**

### 8.3 解决方案

```rust
// 改造后：走 send_msg_to_user() → 本机 + Redis + 跨节点转发
// 复用 process_msg_service.rs 的路由逻辑
```

## 9. 启动模式拆分

### 9.1 现状

当前已有独立二进制入口：

| 入口 | 职责 |
|------|------|

### 9.5 单机模式启动序列（当前）

```rust
// src/main.rs — 当前实现
#[actix_web::main]
async fn main() {
    dotenvy::dotenv().ok();
    init_tracing();

    // 1. QUIC 服务（自包含：加载配置、初始化 Redis/DB、启动 QUIC）
    let _chat_node = start_server().await?;

    // 2. HTTP 服务（再次加载配置、初始化 Redis/DB — 幂等跳过）
    init_server::start_server().await?;
}
```

```rust
const CHAT_TYPE_PRIVATE: u16 = 0;
```

```rust
match text_msg.chat_type {
```
